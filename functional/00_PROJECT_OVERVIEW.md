# Обзор проекта: Конструктор вариантов ЕГЭ по литературе

## Цель проекта

Создать сайт для преподавателя литературы, специализирующегося на подготовке к ЕГЭ в РФ.
**Ключевая фича**: система генерации бесконечного числа уникальных вариантов билетов ЕГЭ на основе "кирпичиков" (отрывков произведений, стихотворений, заданий).

## Основные сущности

- **Вариант ЕГЭ**: сгенерированный набор заданий, соответствующий формату ЕГЭ.
- **Кирпичики**: базовые элементы (отрывки, стихотворения, задания), из которых собираются варианты.
- **Конструктор**: инструмент для генерации вариантов (доступен в меню сайта).
- **Рандомизатор**: серверная логика, отвечающая за случайную генерацию вариантов.

## Проблемы текущей реализации

1. **Перегруженная ручка `api/knowledge-base`**:
   - Огромный объем данных (~5МБ JSON).
   - Пересчет и долгий рендер блокируют главный поток.
2. **Архитектурные проблемы**:
   - Вычисления вариантов происходят на фронтенде (`useMemo` пересчитывается при каждом рендере).
   - Нет кеширования результатов.

## Решение

- Перенести вычисления на сервер (Nuxt 4 в гибридном режиме).
- Кешировать результаты для роутов:
  - `api/knowledge-base` (кеш инвалидируется при изменении базы знаний).
  - `api/variants/runtime/pregenerated` (кеш для общего прегенерированного варианта).
- Использовать Web Worker как fallback для долгих вычислений.

---

## Ключевые требования к агенту

- **Фокус на фронтенд**: Nuxt 4, серверная предзагрузка, оптимизация рендеринга.
- **Не менять бэкенд**: только анализировать payloads для соответствия фронтенда.
- **Минимализм**: не тратить ресурсы на стилизацию, только функционал.
