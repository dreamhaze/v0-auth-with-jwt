# Технические требования

## Стек

- **Фронтенд**: Nuxt 4 (гибридный режим: SSR + SPA).
- **Бэкенд**: не меняем (только анализируем payloads).
- **Кеширование**: серверное (Nitro).
- **Fallback**: Web Worker для долгих вычислений на клиенте.

## Ключевые задачи

1. **Серверная предзагрузка**:
   - Для роутов:
     - `api/knowledge-base`: кешировать ответ, инвалидировать при изменениях в админке.
     - `api/variants/runtime/pregenerated`: кешировать 1 общий вариант для всех пользователей.
2. **Оптимизация рендеринга**:
   - Не блокировать главный поток.
   - При долгих вычислениях (обновление базы) — показывать лоадер.
3. **Страницы**:
   - **Главная**: подсчет количества возможных вариантов (на основе `knowledge-base`).
   - **Страница прегенерированного варианта**: отображение варианта из `/pregenerated`.
   - **Админка**: просмотр всех "кирпичиков" (отрывков, стихотворений, заданий).

## Пример кода для Nuxt 4 (серверная предзагрузка)

```ts
// ~/server/api/knowledge-base.get.ts
export default defineEventHandler(async (event) => {
  const cacheKey = 'knowledge-base';
  const cached = await useStorage().getItem(cacheKey);

  if (cached) return cached;

  // Запрос к бэкенду (или локальный анализ payload)
  const payload = await fetchKnowledgeBasePayload();

  // Кешируем на 1 час (или до инвалидации)
  await useStorage().setItem(cacheKey, payload, { ttl: 3600 });

  return payload;
});
```

## Инвалидация кеша

- **Триггер**: изменение базы знаний в админке. (и по времени ttl 1 час для тестирования)
- **Механизм**:

  - При сохранении изменений в админке — вызывать `POST /api/invalidate-cache`.
  - Удалять кеш для `knowledge-base` и `pregenerated`.
  - Отобразить для тестирования на странице `/admin` разницу в хеше между текущей и предыдущими 5 версиями

- **Fallback**: Web Worker для долгих вычислений на клиенте после запроса (запасной вариант, выключен по умолчанию и не должен влиять на основную реализацию).
